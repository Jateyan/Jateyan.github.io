微前端基础

什么是微前端？它解决了哪些传统前端开发中的问题？（重要）

微前端是一种将前端应用分解成更小、更简单的应用的架构风格，它将微服务的理念应用于浏览器端，即将 Web 应用由单一的单体应用转变为多个小型前端应用聚合为一的应用。各个前端应用还可以独立运行、独立开发、独立部署。解决了下面一系列问题：

  • 巨石应用问题：将巨石应用拆分成多个单体微应用，微应用仓库单独维护，代码独立运行、独立开发、独立部署；
  • 兼容技术栈：主框架不限制接入微应用的技术栈，微应用具备完全自主权；
  • 减小迁移成本：可以低成本地整合已有项目，将已有的项目迁移至新的项目。

请描述微前端架构的主要实现方式及其各自优缺点。（重要）

微前端架构有多种实现方式，主要包括：

  1. 基于iframe的实现

    优点：

    • 天然的JavaScript和CSS隔离
    • 实现简单，接入成本低
    • 各应用间硬隔离，不会互相影响

    缺点：

    • 性能较差，每个iframe都是完整的页面
    • 用户体验不佳（如滚动、历史记录、弹窗等）
    • 通信复杂且受限
    • 难以实现应用间样式统一

  2. 基于路由分发的实现（如single-spa、qiankun）

    优点：

    • 提供统一的路由系统
    • 应用间切换流畅，无刷新
    • 良好的用户体验

    缺点：

    • 需要对应用进行一定改造
    • 可能存在JS/CSS冲突风险
    • 运行时依赖框架提供的沙箱隔离

  3. Web Components实现（如open-wc、micro-app）

    优点：

    • 基于Web标准，兼容性好
    • Shadow DOM提供了原生的样式隔离
    • 可作为真正的组件在页面任意位置使用

    缺点：

    • 老浏览器兼容性问题
    • 学习成本较高
    • 生态系统相对不成熟

  4. Module Federation实现（Webpack 5提供）

    优点：

    • 共享依赖，减少重复加载
    • 应用间无缝集成
    • 支持运行时动态加载

    缺点：

    • 强依赖于Webpack生态
    • 配置相对复杂
    • 缺乏完善的隔离机制

  5. 服务端集成（SSR+ESI等）

    优点：

    • 更好的首屏性能
    • 搜索引擎友好（SEO）
    • 减轻客户端负担

    缺点：

    • 增加服务端复杂度
    • 对后端基础设施要求高
    • 前端开发流程变复杂

微前端架构在哪些业务场景下最适合使用？什么情况下不建议使用？（重要）

答案： 适合使用微前端的场景包含功能复杂、模块众多的大型企业级应用,大型旧系统的渐进式重构；不建议使用微前端的场景：功能简单、规模小的应用，单一团队开发的项目，对性能极其敏感的应用不建议使用微前端场景。

qiankun基础

1. qiankun是什么？它与其他微前端框架相比有什么特点？（重要）

qiankun（乾坤）是蚂蚁金服开源的一个基于single-spa的微前端实现库，旨在帮助大型应用实现微前端架构。qiankun具有以下特点：

1、开箱即用：提供了完整的API和生命周期管理，简化了微前端实施的复杂度；

2、HTML Entry方式： 采用HTML作为应用入口，更接近原生开发体验，简化了静态资源加载和解析；

3、完善的沙箱机制： 提供多种JavaScript沙箱实现，支持多实例沙箱，应用之间完全隔离，CSS隔离方案更加完善；

4、良好的兼容性： 支持主流浏览器和各种前端框架，容各种构建工具和开发环境

5、活跃的社区支持

与其他框架相比：

相比single-spa： qiankun在single-spa基础上增加了HTML Entry、JS沙箱、样式隔离等特性，使用更简单

相比iframe方案： qiankun提供了更好的用户体验和性能表现，同时保持了应用隔离

相比Web Components： qiankun对老旧浏览器兼容性更好，对应用的改造要求更低

相比Module Federation： qiankun提供了更完善的应用隔离方案，不依赖特定构建工具

3. qiankun框架的核心功能有哪些？（重要）

应用加载机制：

  • HTML Entry方式加载微应用
  • 自动解析HTML、提取CSS和JavaScript资源
  • 支持多种加载策略（预加载、按需加载等）

沙箱隔离：

  • JavaScript沙箱：提供多种沙箱实现（快照沙箱、代理沙箱等）
  • CSS隔离：支持Shadow DOM和scoped CSS两种样式隔离方式
  • 全局变量隔离：防止微应用之间的全局变量污染

生命周期管理：

  • 提供标准化的应用生命周期钩子（bootstrap、mount、unmount等）
  • 生命周期函数的自动注册和调用
  • 应用状态管理和错误处理

应用间通信：

  • 基于Props的数据传递
  • 全局状态共享机制
  • 自定义事件系统

路由管理：

  • 基于路由的应用激活和切换
  • 支持各种路由模式（hash、history）
  • 路由劫持和转发

预加载策略：

  • 支持应用资源预加载
  • 可配置的预加载策略
  • 智能的资源加载优化

插件机制：

  • 提供可扩展的插件API
  • 支持全局插件和应用级插件
  • 自定义加载过程和运行时行为

4. 为什么qiankun选择基于single-spa进行开发？它相比single-spa增强了哪些能力？（重要）

选择基于single-spa开发的原因：

  1. 成熟的基础设施：

    • single-spa是最早的微前端框架之一，有成熟的架构设计
    • 提供了可靠的应用注册、激活和生命周期管理基础设施
    • 社区活跃，生态系统完善

  2. 专注核心问题：

    • single-spa解决了应用注册、路由劫持、生命周期管理等核心问题
    • 这让qiankun可以专注于解决中国开发者面临的特定挑战

  3. 避免重复造轮子：

    • 利用现有成熟方案作为基础可以加速开发
    • 可以借鉴single-spa的成功经验

  4. 扩展而非替代：

    • 采用扩展的方式而非完全重写，降低了开发风险
    • 保持了与single-spa生态的兼容性

qiankun相比single-spa增强的能力：

  1. HTML Entry加载方式：

    • single-spa只支持JavaScript Entry
    • qiankun创新性地支持HTML作为入口，更符合开发习惯
    • 简化了资源加载和解析过程

  2. 完善的沙箱机制：

    • single-spa没有内置的沙箱隔离机制
    • qiankun提供了多种JavaScript沙箱实现
    • 增加了对不同浏览器的兼容性支持

  3. CSS隔离方案：

    • single-spa不处理样式隔离问题
    • qiankun提供了Shadow DOM和scoped CSS两种隔离方案
    • 有效防止了样式冲突

  4. 更完善的应用通信：

    • 提供了全局状态管理机制
    • 简化了应用间通信

  5. 资源预加载：

    • 支持智能的应用预加载策略
    • 优化了应用加载性能

  6. 更好的开发体验：

    • 提供了更友好的API和配置选项
    • 简化了接入流程
    • 提供了中文文档和社区支持

  7. 兼容性处理：

    • 更好地支持IE等传统浏览器
    • 处理了各种浏览器兼容性问题

5. qiankun中的应用如何注册和加载？基本配置项有哪些？

qiankun中应用的注册和加载流程(重要)：

  1. 注册微应用：

    • 在主应用中调用registerMicroApps方法注册微应用
    • 或使用loadMicroApp手动加载微应用

  2. 启动qiankun：

    • 调用start方法启动qiankun框架

  3. 匹配路由：

    • qiankun根据当前URL和应用的activeRule判断是否激活应用

  4. 加载应用资源：

    • 从应用的entry获取HTML内容
    • 解析HTML，提取JavaScript、CSS等资源

  5. 创建沙箱环境：

    • 为应用创建JavaScript沙箱
    • 设置CSS隔离

  6. 执行应用代码：

    • 在沙箱环境中执行应用代码
    • 调用应用的生命周期函数

基本配置项：

registerMicroApps([
  {
    name: 'app1', // 应用名称，唯一标识
    entry: '//localhost:8080', // 应用入口，HTML地址
    container: '#container', // 应用容器选择器
    activeRule: '/app1', // 激活规则，通常为URL前缀
    props: { // 传递给微应用的数据
      data: {
        publicPath: '//localhost:8080'
      },
      methods: {
        onGlobalStateChange: () => {},
        setGlobalState: () => {}
      }
    }
  }
]);

主要配置项说明：

  1. name：

    • 应用的唯一标识
    • 用于区分不同的微应用

  2. entry：

    • 微应用的入口地址
    • 可以是完整URL或相对路径
    • qiankun会从这个地址加载HTML内容

  3. container：

    • 微应用的挂载点
    • CSS选择器或DOM元素
    • 微应用将在此处渲染

  4. activeRule：

    • 应用的激活规则
    • 可以是字符串（URL前缀）
    • 也可以是函数（返回布尔值判断是否激活）

  5. props：

    • 传递给微应用的数据和方法
    • 可包含普通数据、回调函数等
    • 微应用通过生命周期函数接收

start方法配置项：

start({
  prefetch: true, // 是否预加载
  sandbox: true, // 是否启用沙箱
  singular: true, // 是否单实例模式
  fetch: () => {}, // 自定义fetch方法
  getPublicPath: () => {}, // 获取publicPath的方法
  excludeAssetFilter: () => false // 排除特定资源的过滤函数
});

主要配置选项包括是否开启预加载、沙箱隔离、单实例模式等，以及各种自定义处理函数。

技术实现篇

应用加载与生命周期

1. 详细解释qiankun的JS沙箱机制，它是如何实现的？（重要）

qiankun提供了多种JS沙箱实现，用于隔离微应用的JavaScript运行环境，防止应用间互相干扰。主要包括以下几种沙箱实现：

  1. 快照沙箱（SnapshotSandbox）：

    • 工作原理：在应用切换时记录和恢复全局状态
    • 实现方式：
      class SnapshotSandbox {
  constructor() {
    this.windowSnapshot = {};
    this.modifyPropsMap = {};
  }
  
  // 激活沙箱
  active() {
    // 1. 记录当前window对象的状态
    this.windowSnapshot = {...window};
    
    // 2. 恢复上次修改的状态
    Object.keys(this.modifyPropsMap).forEach(prop => {
      window[prop] = this.modifyPropsMap[prop];
    });
  }
  
  // 失活沙箱
  inactive() {
    // 1. 记录当前被修改的属性
    for (const prop in window) {
      if (window[prop] !== this.windowSnapshot[prop]) {
        this.modifyPropsMap[prop] = window[prop];
        // 2. 恢复window原始状态
        window[prop] = this.windowSnapshot[prop];
      }
    }
  }
}
    • 适用场景：单实例模式，不支持多个应用同时运行
    • 兼容性：支持所有浏览器，包括IE

  2. 代理沙箱（ProxySandbox）：

    • 工作原理：使用ES6 Proxy拦截对全局对象的访问和修改
    • 实现方式：
      class ProxySandbox {
  constructor() {
    this.updateValueMap = new Map();
    
    const fakeWindow = {};
    const proxy = new Proxy(fakeWindow, {
      // 获取属性时的处理
      get: (target, prop) => {
        return this.updateValueMap.has(prop) 
          ? this.updateValueMap.get(prop) 
          : window[prop];
      },
      
      // 设置属性时的处理
      set: (target, prop, value) => {
        this.updateValueMap.set(prop, value);
        return true;
      }
    });
    
    this.proxy = proxy;
  }
  
  // 激活沙箱
  active() {
    // 无需特殊处理，代理一直生效
  }
  
  // 失活沙箱
  inactive() {
    // 无需特殊处理，应用可以保持自己的状态
  }
}
    • 适用场景：多实例模式，支持多个应用同时运行
    • 兼容性：需要支持ES6 Proxy的现代浏览器

  3. 遗留代理沙箱（LegacySandbox）：

    • 工作原理：结合快照和代理的特点，用于兼容旧浏览器
    • 实现方式：通过属性劫持和状态记录相结合
    • 适用场景：需要同时支持现代和传统浏览器的环境

qiankun根据浏览器环境和配置自动选择最适合的沙箱实现：

  • 支持Proxy的现代浏览器使用ProxySandbox
  • 不支持Proxy但需要多实例的使用LegacySandbox
  • 单实例模式下可以使用SnapshotSandbox

沙箱在微应用加载时激活，在卸载时失活，确保全局环境的干净和隔离。

2. qiankun的CSS隔离是如何实现的？有哪些模式？ （重要）

qiankun提供了两种CSS隔离模式，用于防止微应用之间的样式冲突：

  1. 严格隔离模式（Shadow DOM）：

    实现原理：

    • 利用浏览器原生的Shadow DOM机制
    • 为每个微应用创建一个独立的Shadow DOM
    • 应用的所有DOM操作和样式都在Shadow DOM内部生效

    实现方式：

    // 创建Shadow DOM容器
const shadow = container.attachShadow({ mode: 'open' });

// 将微应用的内容挂载到Shadow DOM中
const mountDOM = document.createElement('div');
shadow.appendChild(mountDOM);

// 样式完全隔离在Shadow DOM内部

    优点：

    • 完全的样式隔离，几乎不可能发生样式泄漏
    • 基于浏览器原生机制，性能较好
    • 最彻底的CSS隔离方案

    缺点：

    • 一些老旧浏览器不支持Shadow DOM
    • 可能影响一些UI库的正常工作
    • 弹窗类组件可能无法正确显示

  2. 宽松隔离模式（Scoped CSS）：

    实现原理：

    • 通过动态添加选择器前缀的方式实现样式隔离
    • 为每个微应用的样式添加唯一的命名空间
    • 所有CSS规则都会被转换为带前缀的形式

    实现方式：

    // 样式处理函数
function scopedCSS(styleElement, appName) {
  const cssText = styleElement.textContent;
  // 转换CSS规则，添加命名空间前缀
  const scopedCssText = cssText.replace(
    /([^\r\n,{}:]+)(,(?=[^}]*{)|\s*{)/g,
    (match, selector, suffix) => {
      // 为每个选择器添加命名空间前缀
      return selector
        .split(',')
        .map(s => `div[data-qiankun="${appName}"] ${s.trim()}`)
        .join(',') + suffix;
    }
  );
  
  // 替换原始样式内容
  styleElement.textContent = scopedCssText;
}

    优点：

    • 兼容性好，适用于所有浏览器
    • 对UI库的兼容性更好
    • 弹窗、悬浮层等元素可正常工作

    缺点：

    • 隔离效果不如Shadow DOM彻底
    • 可能存在样式泄漏的风险
    • 动态添加的样式需要额外处理

配置方式：

通过start方法的sandbox选项配置CSS隔离模式：

// 配置样式隔离模式
start({
  sandbox: {
    // 是否开启严格样式隔离
    strictStyleIsolation: false,
    // 是否开启宽松样式隔离
    experimentalStyleIsolation: true
  }
});

在实际应用中，qiankun会根据配置和浏览器环境选择合适的CSS隔离方式。大多数情况下，宽松隔离模式（experimentalStyleIsolation）是更实用的选择，它在隔离效果和兼容性之间取得了较好的平衡。

3. qiankun如何处理子应用的生命周期？各生命周期函数的作用是什么？（重要）

qiankun基于single-spa的生命周期机制，为微应用定义了标准化的生命周期：

主要生命周期函数：

  1. bootstrap：

    • 触发时机：微应用初始化时调用，只会调用一次
    • 用途：应用初始化工作，如配置、预加载资源等
    • 示例：
      export async function bootstrap() {
  console.log('应用初始化');
  // 加载配置、初始化SDK等
}

  2. mount：

    • 触发时机：微应用每次挂载时调用
    • 用途：创建应用实例，将应用渲染到指定容器
    • 接收参数：props（包含container和传入的数据）
    • 示例：
      export async function mount(props) {
  console.log('应用挂载');
  const { container, data } = props;
  // 渲染应用到container
  ReactDOM.render(<App {...data} />, container.querySelector('#root'));
}

  3. unmount：

    • 触发时机：微应用每次卸载时调用
    • 用途：清理应用资源，移除DOM，防止内存泄漏
    • 接收参数：props（包含container等信息）
    • 示例：
      export async function unmount(props) {
  console.log('应用卸载');
  const { container } = props;
  // 卸载应用实例
  ReactDOM.unmountComponentAtNode(container.querySelector('#root'));
}

  4. update (可选)：

    • 触发时机：微应用更新时调用（如props变化）
    • 用途：响应配置变更，更新应用状态
    • 接收参数：props（更新后的props）
    • 示例：
      export async function update(props) {
  console.log('应用更新', props);
  // 响应props变化，更新应用状态
}

生命周期调用流程：

应用首次加载: bootstrap → mount
应用切换: unmount(上一个应用) → mount(下一个应用)
应用属性更新: update
应用卸载: unmount

qiankun对生命周期的处理：

  1. 生命周期注册：

    • 通过registerMicroApps方法注册微应用的生命周期
    • 框架自动管理生命周期调用时机

  2. 生命周期增强：

    • 在调用应用生命周期前后添加框架逻辑
    • 如在mount前激活沙箱，在unmount后失活沙箱

  3. 错误处理：

    • 为每个生命周期函数添加错误处理
    • 防止一个应用的错误影响其他应用

  4. 异步支持：

    • 所有生命周期函数都支持异步操作（Promise）
    • 确保各阶段任务完成后再进入下一阶段

props传递：

qiankun会向生命周期函数传递props，包含：

  • container：微应用的容器元素
  • name：微应用的名称
  • window：微应用的沙箱window对象
  • 用户自定义的数据和方法

通过这种标准化的生命周期管理，qiankun确保了微应用的可预测行为和良好的资源管理。

4. HTML Entry方式加载应用的原理是什么？相比JS Entry有哪些优势？（重要）

答案：

HTML Entry加载原理：

HTML Entry是qiankun的一大创新，它通过直接加载微应用的HTML入口文件来实现应用加载。工作流程如下：

  1. 获取HTML内容：

    • 通过fetch请求获取入口HTML文件内容
    async function importHTML(url) {
  const html = await fetch(url).then(response => response.text());
  return processHTML(html, url);
}

  2. 解析HTML：

    • 解析HTML文档，提取其中的scripts、styles、body等内容
    function processHTML(html, baseUrl) {
  const doc = new DOMParser().parseFromString(html, 'text/html');
  
  // 提取脚本元素
  const scripts = Array.from(doc.querySelectorAll('script')).map(script => {
    const src = script.getAttribute('src');
    return { src, content: script.textContent, async: script.async, ... };
  });
  
  // 提取样式元素
  const styles = Array.from(doc.querySelectorAll('style, link[rel="stylesheet"]')).map(...);
  
  // 提取页面内容
  const body = doc.querySelector('body').innerHTML;
  
  return { scripts, styles, body };
}

  3. 加载外部资源：

    • 对于外部脚本和样式，通过fetch加载其内容
    async function loadScripts(scripts, baseUrl) {
  return Promise.all(scripts.map(async script => {
    if (script.src) {
      script.content = await fetch(resolveUrl(script.src, baseUrl))
        .then(response => response.text());
    }
    return script;
  }));
}

  4. 执行脚本：

    • 在沙箱环境中执行所有JavaScript脚本
    function executeScripts(scripts, proxy) {
  const executor = window.eval;
  
  scripts.forEach(script => {
    // 使用with将全局变量指向代理对象
    const code = `with(window) { ${script.content} }`;
    executor.call(proxy, code);
  });
}

  5. 应用样式：

    • 将样式应用到微应用容器中
    function appendStyles(styles, container) {
  styles.forEach(style => {
    const styleElement = document.createElement('style');
    styleElement.textContent = style.content;
    container.appendChild(styleElement);
  });
}

  6. 挂载DOM：

    • 将HTML内容挂载到容器中
    function mount(body, container) {
  container.innerHTML = body;
}

相比JS Entry的优势：

  1. 更接近原生开发体验：

    • 开发者可以使用熟悉的HTML入口文件
    • 无需特殊的构建配置或导出格式
    • 与传统SPA开发方式保持一致

  2. 资源加载更完整：

    • 自动处理HTML中声明的所有资源
    • 包括内联脚本、样式和外部资源
    • 更好地支持动态加载的scripts和样式

  3. 简化配置：

    • 只需配置HTML入口地址
    • 无需手动指定各个资源文件
    • 减少了遗漏资源的可能性

  4. 更好的兼容性：

    • 对各种构建工具的依赖更少
    • 更容易集成现有应用
    • 支持各种资源加载方式

  5. 动态资源处理：

    • 能处理动态生成的script标签
    • 支持按需加载的资源
    • 更符合现代前端应用的资源加载模式

  6. 更少的改造成本：

    • 微应用改造更简单
    • 不需要特殊的打包配置
    • 降低了接入门槛

HTML Entry方式是qiankun的重要创新，它显著提升了微前端的开发体验和易用性，让微应用的接入更加自然和顺畅。

通信与状态管理

1. qiankun中有哪些应用间通信方式？各有什么特点？(重要)

qiankun提供了多种应用间通信方式，满足不同场景的需求：

  1. 基于Props的通信方式

    原理：主应用通过props将数据和方法传递给微应用。

    实现方式：

    // 主应用注册微应用时传递props
registerMicroApps([
  {
    name: 'app1',
    entry: '//localhost:8080',
    container: '#container',
    activeRule: '/app1',
    props: {
      data: { userName: 'qiankun' },
      methods: { logout: () => console.log('logout') }
    }
  }
]);

// 微应用在生命周期函数中接收props
export function mount(props) {
  console.log(props.data.userName); // 'qiankun'
  props.methods.logout(); // 调用主应用方法
}

    特点：

    • 简单直接，类似React组件通信
    • 适合主应用向微应用传递数据和方法
    • 微应用只能与主应用通信，无法直接与其他微应用通信
    • 单向数据流，主应用控制数据传递

  2. 基于全局状态的通信方式（GlobalState）

    原理：qiankun提供了全局状态管理机制，所有应用共享同一个状态对象。

    实现方式：

    // 主应用初始化全局状态
import { initGlobalState } from 'qiankun';

// 初始化状态
const actions = initGlobalState({ user: 'qiankun' });

// 监听状态变化
actions.onGlobalStateChange((state, prev) => {
  console.log('主应用监听到状态变化：', state, prev);
});

// 修改状态
actions.setGlobalState({ user: 'newUser' });

// 微应用中使用全局状态
export function mount(props) {
  // 获取通信方法
  const { onGlobalStateChange, setGlobalState } = props;
  
  // 监听状态变化
  onGlobalStateChange((state, prev) => {
    console.log('微应用监听到状态变化：', state, prev);
  });
  
  // 修改状态
  setGlobalState({ count: 1 });
}

    特点：

    • 类似全局状态管理（如Redux）
    • 支持多应用之间相互通信
    • 状态变更通知机制
    • 适合复杂数据流场景

  3. 自定义事件通信方式

    原理：基于浏览器的自定义事件机制实现发布-订阅模式。

    实现方式：

    // 封装事件总线
class EventBus {
  on(event, callback) {
    window.addEventListener(event, callback);
  }
  
  emit(event, data) {
    window.dispatchEvent(new CustomEvent(event, { detail: data }));
  }
  
  off(event, callback) {
    window.removeEventListener(event, callback);
  }
}

// 创建事件总线实例
const eventBus = new EventBus();

// 主应用使用
eventBus.on('event-from-micro-app', (event) => {
  console.log('收到微应用消息：', event.detail);
});

// 微应用使用
eventBus.emit('event-from-micro-app', { data: 'hello' });

    特点：

    • 松耦合，事件发布者和订阅者互不依赖
    • 支持多对多通信
    • 无需依赖qiankun提供的API
    • 适合事件驱动的场景

  4. 基于共享库的通信方式

    原理：通过共享第三方库（如rxjs）在应用间建立通信。

    实现方式：

    // 共享模块（如在公共依赖中）
import { Subject } from 'rxjs';

// 创建消息总线
export const messageBus = new Subject();

// 在主应用中
import { messageBus } from 'shared-lib';

// 订阅消息
messageBus.subscribe(message => {
  console.log('主应用收到消息：', message);
});

// 在微应用中
import { messageBus } from 'shared-lib';

// 发送消息
messageBus.next({ from: 'microApp', data: 'hello' });

    特点：

    • 功能强大，可利用第三方库的全部特性
    • 需要主应用和微应用共享依赖
    • 适合有复杂数据流处理需求的场景
    • 通信逻辑可以更加结构化

  5. LocalStorage/SessionStorage通信

    原理：利用浏览器存储事件在应用间共享数据。

    实现方式：

    // 应用A：设置数据
localStorage.setItem('shared-data', JSON.stringify({ message: 'hello' }));

// 应用B：监听storage事件
window.addEventListener('storage', event => {
  if (event.key === 'shared-data') {
    const data = JSON.parse(event.newValue);
    console.log('收到共享数据：', data);
  }
});

    特点：

    • 简单易用，无需额外库
    • 适合不同标签页/窗口间通信
    • 存储空间有限
    • 只能传递可序列化的数据

选择通信方式的考虑因素：

  1. 通信复杂度：简单场景可选Props传递，复杂场景选GlobalState或自定义事件

  2. 通信方向：

    • 主应用→微应用：Props最简单
    • 微应用→主应用：GlobalState或自定义事件
    • 微应用↔微应用：GlobalState或自定义事件最适合

  3. 耦合程度：低耦合需求选择自定义事件，紧密协作可用GlobalState

  4. 性能要求：频繁通信选择自定义事件或共享库方案

在实际应用中，可以组合使用多种通信方式以满足不同需求。例如，使用Props传递初始数据和方法，使用GlobalState处理共享状态，用自定义事件处理一次性通知。

常见问题与解决方案

1. 使用qiankun过程中常见的跨域问题及解决方案？

答案：

在qiankun微前端架构中，跨域问题是最常见的挑战之一，主要表现在以下几个方面：

1. 资源加载跨域

  • 问题表现：主应用无法加载微应用的入口HTML、JS或CSS资源，控制台报错：

    Access to fetch at 'http://app1.example.com/index.html' from origin 'http://main.example.com' has been blocked by CORS policy

  • 解决方案：

    1. 设置正确的CORS头：

      // 微应用服务器设置
Access-Control-Allow-Origin: http://main.example.com
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Headers: Content-Type

    2. 使用开发代理：

      // webpack配置
devServer: {
  proxy: {
    '/micro-app1': {
      target: 'http://app1.example.com',
      changeOrigin: true,
      pathRewrite: { '^/micro-app1': '' }
    }
  }
}

    3. 部署到同一域：在生产环境将所有应用部署到同一个域下的不同路径

2. 接口请求跨域

  • 问题表现：微应用内部的Ajax请求跨域

  • 解决方案：

    1. 微应用接口代理：

      // 主应用服务器配置统一的接口代理
location /api/ {
  proxy_pass http://api.example.com/;
  proxy_set_header Host api.example.com;
}

    2. 使用公共认证：微应用接口和主应用使用同一个认证服务

    3. 注入统一请求库：

      // 主应用提供请求方法
const request = {/*...*/};

// 注册时传入
registerMicroApps([{
  name: 'app1',
  entry: '//localhost:8080',
  container: '#container',
  props: { request }
}]);

3. Cookie跨域问题

  • 问题表现：主域名下Cookie无法被子域名应用访问或使用

  • 解决方案：

    1. 设置Cookie的domain：

      // 设置共享Cookie
document.cookie = "token=123; domain=.example.com";

    2. 采用统一身份认证：使用JWT等token认证方式代替Cookie

    3. 通过props传递认证信息：

      // 主应用
const token = getToken();
registerMicroApps([{
  name: 'app1',
  props: { token }
}]);

4. localStorage/sessionStorage隔离问题

  • 问题表现：不同域下的应用无法共享localStorage

  • 解决方案：

    1. 使用qiankun的全局状态共享：

      // 主应用
const actions = initGlobalState({ token: localStorage.getItem('token') });

// 监听变化并同步到localStorage
actions.onGlobalStateChange((state) => {
  if (state.token) localStorage.setItem('token', state.token);
});

    2. iframe postMessage通信：

      // 发送消息
window.parent.postMessage({ key: 'token', value: 'xxx' }, '*');

// 接收消息
window.addEventListener('message', (e) => {
  if (e.data.key === 'token') {
    localStorage.setItem('token', e.data.value);
  }
});

5. 开发环境与生产环境差异

  • 问题表现：开发环境工作正常，生产环境出现跨域问题

  • 解决方案：

    1. 统一环境配置：确保开发和生产环境的域名策略一致
    2. 使用环境变量：根据环境动态设置应用entry地址
      registerMicroApps([{
  name: 'app1',
  entry: process.env.NODE_ENV === 'production' 
    ? '//app1.example.com' 
    : '//localhost:8080'
}]);

最佳实践总结：

  1. 同域部署：生产环境尽量将主应用和微应用部署在同一域名下不同路径
  2. 统一后端网关：所有API请求通过同一个网关，避免跨域
  3. 权限统一管理：实现统一的认证授权中心
  4. 本地开发代理：开发环境通过代理解决跨域问题
  5. 合理使用qiankun通信机制：优先使用qiankun提供的通信方式共享数据

通过以上方法，可以有效解决qiankun微前端架构中遇到的各种跨域问题。

2. qiankun框架实现微前端的优缺点有哪些？

答案：

qiankun框架的优点：

  1. 开箱即用，配置简单：

    • 提供完善的API设计，使用门槛低
    • 快速集成，无需复杂配置
    • 文档完善，中文支持好

  2. HTML Entry加载方式：

    • 直接使用HTML作为应用入口
    • 更接近原生开发体验
    • 无需特殊的构建配置

  3. 完善的沙箱隔离机制：

    • 提供多种JavaScript沙箱实现
    • CSS隔离方案更加完善
    • 有效防止应用间互相影响

  4. 生态成熟，社区活跃：

    • 蚂蚁金服官方维护
    • 大量的实际应用案例
    • 问题解决方案丰富

  5. 良好的兼容性：

    • 支持各种主流浏览器
    • 兼容不同的前端框架
    • 对老旧系统有较好的兼容性

  6. 丰富的通信机制：

    • 提供全局状态管理
    • 支持多种应用间通信方式
    • 数据共享能力强

  7. 路由管理能力：

    • 支持基于路由的应用加载
    • 可配置的预加载策略
    • 应用间无缝跳转

qiankun框架的缺点：

  1. 性能开销：

    • 沙箱机制带来额外的性能消耗
    • 应用加载和切换可能有延迟
    • 首次加载资源较多

  2. 学习曲线：

    • 需要理解微前端的基本原理
    • 沙箱、生命周期等概念需要学习
    • 问题排查需要一定的经验

  3. 调试复杂性：

    • 多应用联调带来困难
    • 错误定位比单体应用复杂
    • 开发工具链不够完善

  4. SSR支持有限：

    • 对服务端渲染支持不完善
    • SEO能力有限
    • 不适合内容类网站

  5. 样式隔离不彻底：

    • 非Shadow DOM模式下可能有样式泄漏
    • 复杂UI库可能出现样式问题
    • 需要额外的规范约束

  6. 子应用需要一定改造：

    • 子应用需要导出生命周期函数
    • 路由需要考虑基础路径问题
    • 全局事件和定时器需要特殊处理

  7. 部分场景兼容性问题：

    • 一些特殊API在沙箱中可能受限
    • 动态挂载的组件可能逃离沙箱
    • iframe嵌套场景处理复杂

适用场景选择：

qiankun特别适合以下场景：

  • 大型企业级应用拆分
  • 多团队协作的系统
  • 技术栈渐进式升级
  • 系统整合与遗留系统改造

不太适合的场景：

  • 对性能极其敏感的应用
  • 简单小型应用
  • 高度定制化的前端框架
  • 需要深度SSR支持的内容网站

使用建议：

  1. 评估适用性：根据项目特点判断是否需要微前端架构
  2. 循序渐进：微前端实施应该分阶段进行
  3. 技术规范：建立统一的开发规范和公共库
  4. 合理拆分：按业务领域而非技术边界拆分应用
  5. 性能优化：关注首屏加载和应用预加载策略

通过合理评估和规划，可以充分发挥qiankun的优势，规避其缺点带来的影响。
