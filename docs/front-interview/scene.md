## 1、如何在浏览器中执行100万个任务，并保证页面不卡顿？

**思路：** 任务分批做处理，避免主线程阻塞
直接在浏览器主线程一次性执行100万个任务，会导致页面卡顿甚至无响应。常用的做法是将任待下一帧或下一个空闲时间再继续。
**常用方案**
- requestAnimationFrame:每一帧处理一部分任务,适合动画、渲染相关任务，每帧执行，和页面刷新同步。 
- setTimeout/setInterval:通过定时器分批处理任务，间隔执行，避免长时间占用主线程,适合通用分批任务，兼容性好，但精度较低。
- MessageChanne:利用微任务队列，分批调度任务，性能优于setTimeout,适合需要更高精度的分批任务。
- requestldleCallback:浏览器空闲时才执行任务，适合低优先级、可延迟的任务，只有浏览器空闲时才执行。

优化方案:
- 大表单渲染优化:基于React调度器思想的时间分片机制，实现高性能表单渲染。通过动态帧率自适应算法精确计算执行预算，结合requestAnimationFrame同步浏览器渲染周期，将复杂渲染任务拆分为微批次执行。巧妙运用NgZone边界管理，在区域外协调任务调度，仅在实际DOM更新时回到Angular上下文，有效减少变更检测开销，实现无阻塞的渐进式渲染策略。
- 图片懒加载：采用IntersectionObserver API实现高效视口检测，通过零轮询的观察者模式替代传统滚动事件监听。将观察器实例化和元素跟踪操作置于Angular区域外执行，仅在元素进入可视区域时切换至Angular上下文触发资源加载，最大限度减少不必要的变更检测周期，实现对内存和CPU的双重优化。

## 3、谈谈IntersectionObserver

IntersectionObserver是一个浏览器API，用于异步观察目标元素与祖先元素或视口(viewport)的交叉状态变化。它解决了传统滚动事件监听带来的性能问题，提供了一种高效、异步的元素可见性检测机制。

**核心工作原理**
- 1、观察者(Observer)注册目标元素
- 2、目标元素与视口交叉状态改变时触发回调
- 3、浏览器优化执行时机，通常在主线程空闲时执行
- 4、完全异步操作，不阻塞主线程

**主要应用场景**
- 图片懒加载：图片进如视图区域内才将src替换为真正的地址
```javascript
const imgObserver = new IntersectionObserver((entries, observer) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      const src = img.getAttribute('data-src');
      
      if (src) {
        img.setAttribute('src', src);
        img.removeAttribute('data-src');
      }
      
      // 图片加载后取消观察
      observer.unobserve(img);
    }
  });
});

// 为所有懒加载图片注册观察
document.querySelectorAll('img[data-src]').forEach(img => {
  imgObserver.observe(img);
});
```
- 无限滚动：观察加载更多的div是否进入到视口范围内，来确定是否加载下一页数据

```javascript
const loadMoreObserver = new IntersectionObserver((entries) => {

loadMoreObserver.observe(document.querySelector('.load-more-sentinel'));
```

## 4、谈谈 MutationObserver？
MutationObserver是一个浏览器API，用于监听DOM树的变化并做出响应。它能够异步观察DOM元素，监听其子节点、属性、文本内容等变化，当DOM发生变化时触发注册的回调函数。
MutationObserver采用异步回调机制，与IntersectionObserver类似，它不会阻塞主线程，而是在浏览器空闲时批量处理DOM变化。

实际运用场景： 通过MutationObserver建通富文本渲染加载情况，确定是否显示加载更多；

## 6、vue的响应式原理是什么？angular的响应式原理是什么？react的响应式原理是什么？
**vue的响应式原理** 

- Vue2在初始化数据时，会使用Object.defineProperty重新定义data中的所有属性，当页面使用对应属性时，首先会进行依赖收集(收集当前组件的watcher)如果属性发生变化会通知相关依赖进行更新操作(发布订阅)（详细步骤: Vue2采用Object.defineProperty实现数据劫持和发布订阅模式。初始化阶段，Vue实例创建时，对data中所有属性使用Object.defineProperty重新定义，为每个属性创建专门的依赖收集器(Dep).依赖收集阶段，当组件渲染时，使用的data属性会触发getter，getter中会将当前组件的Watcher添加到该属性的依赖收集器中。更新触发的时候，当属性值发生变化时，会触发setter，setter中通知依赖收集器更新所有收集的Watcher，Watcher触发组件重新渲染）。
- Vue3使用Proxy取代Object.defineProperty，直接监听整个对象而非属性，可监听数组变化，可拦截多达13种操作，如删除属性、新增属性等，性能更好且是现代浏览器标准。Vue3通过递归方式实现深度响应式：当Reflect.get返回的值为对象时，再通过reactive方法做代理。通过判断key是否为对象自身属性，比较新旧值是否相等避免不必要的触发，避免数组操作触发多次get/set。

**angular的响应式原理**

- 1、Angular的响应式原理基于变更检测(Change Detection)机制，主要依靠Zone.js监听异步事件触发检测，而非像Vue那样直接监听数据变化。
- 2、Angular采用"脏检测"方式实现响应式,当异步操作（如用户点击、HTTP请求、定时器等）完成时，Angular会从组件树根部开始检查所有绑定是否变化。脏检测机制简单可靠，但在组件树庞大时会带来性能问题，因此Angular提供了多种策略来优化检测过程。比如 OnPush策略，手动分离检测器（detectChanges），运行区域外执行，不可变数据，纯管道等等。
- 3、为什么称为"脏检测": "脏"表示数据已改变需要更新视图。传统脏检测会对所有绑定数据进行检查，无论它是否可能发生变化，这种简单但全面的检查方式被称为脏检测。

**react的响应式原理**

React不同于Vue和Angular，并非采用真正的"响应式"系统，而是基于"单向数据流"和"显式更新"机制：React通过setState/useState API显式通知框架状态已变化，这些API会标记组件为"需要更新"，加入更新队列。
-  多次setState → 合并到更新队列 → 调度系统按优先级处理 → 一次性更新DOM 通过Fiber架构实现时间切片和任务优先级，使渲染可中断和恢复。


## 7、ref和reactive的区别？
- 数据包装方式：reactive: 使用Proxy直接代理整个对象，无需额外包装；ref: 创建包含value属性的对象，通过类的getter/setter拦截.value的访问
- 原始类型处理：reactive: 无法处理原始类型，必须是对象；ref: 通过RefImpl类包装原始值，实现原始类型的响应式
- 依赖收集差异： reactive: 每个属性有独立的依赖集合，粒度更细，ref: 整个ref对象只有一个依赖集合，粒度较粗
- 嵌套处理机制：reactive递归拦截嵌套对象，属性访问时自动深度代理；ref值为对象时内部调用reactive，value获取/设置时自动处理深层代理


## 8、谈谈react的Fiber架构？（略）
Fiber架构是React 16引入的重构核心算法，解决了大型React应用中的主要问题：
- 解决长任务阻塞：传统递归渲染无法中断，导致主线程长时间占用
- 实现任务优先级：不同更新有不同紧急程度
- 支持并发模式：同时处理多个任务

Fiber核心是一种特殊的链表树结构，每个节点包含三个关键引用（第一个子节点，兄弟节点，父节点）：这种结构支持深度优先遍历且可随时中断和恢复，使用迭代而非递归实现。


## 9、什么是虚拟DOM，什么是diff算法?
-虚拟DOM就是用JavaScript对象模拟真实DOM结构树；虚拟DOM通过三个核心步骤工作：
- 当数据变化时，生成新的虚拟DOM树
- 对比新旧虚拟DOM树，找出变更；
- 只更新真实DOM中变化的部分

这种机制避免了频繁直接操作DOM，提高了性能。

**Diff算法**

Diff算法是虚拟DOM的核心，负责比较新旧虚拟DOM树，找出最小变更集。算法核心都采用三个基本假设：
- 不同类型的元素产生不同的树
- 开发者可通过key属性标识哪些元素在不同渲染中保持稳定
- 只对同级元素进行比较

Vue2采用双端比较算法，同时从新旧节点的两端向中间比较，找到可复用节点则移动，否则创建/删除。Vue3在此基础上添加静态标记(PatchFlag)，跳过静态内容，静态提升，复用不变节点，引入Fragment，减少嵌套层级


React使用单向遍历算法，
- 第一次遍历：同位置比较，尝试复用
- 第二次遍历：处理新增节点
- 第三次遍历：删除多余节点

React 16+使用Fiber架构：
- 将比对过程分为多个小任务
- 可暂停、继续、放弃更新
- 配合优先级调度系统

优化特点：
- Vue3编译时标记静态节点,基于依赖追踪，只有依赖变化时才触发更新,更新粒度更细，可精确到属性级别
- react基于Fiber架构实现时间切片，高优先级更新可打断低优先级更新，配合memo/useMemo/shouldComponentUpdate手动优化；

## 10、vue,react和angular的区别是什么？
- 核心设计理念：vue是一个渐进式框架，易学易用，灵活渐进采用；react是一个声明式UI库，通过组件化，状态驱动视图；angular提供了完整解决方案，企业级应用；
- 响应式原理方面：Vue是基于Proxy(Vue3)/Object.defineProperty(Vue2)的数据劫持，实现细粒度响应式；React是基于显式setState/useState触发更新，通过虚拟DOM比对确定变化；Angular是基于Zone.js变更检测，脏检查机制追踪变化；
- 模板与渲染：vue基于HTML的模板语法实现,支持单文件组件;React采用JSX语法，一切皆JavaScript；angular基于TypeScript,增强HTML模板
- 组件通信:Vue采用props/emit、provide/inject、EventBus、Vuex/Pinia；React采用props、Context、Redux/MobX、回调函数等；Angular采用@Input/@Output、服务注入、RxJS、NgRx等
- 性能优化：Vue基于proxy进行细粒度依赖追踪，组件缓存keep-alive；react基于fiber架构实现并发模式;Angular采用AOT编译，OnPush变更检测策略，纯管道
- 生态与工具链：Vue有 Vue Router,Vuex/Pinia,Vue CLI/Vite等

## 15、什么是AOT编译？
AOT编译指的是在程序运行前将代码预先编译成目标格式的技术，而不是在运行时执行编译。
- Vue：模板预编译
- Svelte：编译时转换为高效JavaScript
- React：Prepack等工具提供部分AOT能力
AOT编译的优势：
- 更快的启动时间：无需运行时编译
- 更小的应用体积：不需要包含编译器
- 提前发现模板错误：编译期检查
- 更好的安全性：避免模板注入攻击风险
- 更好的性能：优化编译结果

AOT编译的劣势
- 编译时间较长：增加构建时间
- 缺少动态性：难以处理完全动态的代码
- 调试相对困难：代码转换后不直观
- 构建复杂度增加：需要额外配置
## 16、vue中的spa应用如何优化首屏加载速度？ 
- 请求优化：CDN 将第三方的类库放到 CDN 上，能够大幅度减少生产环境中的项目体积，另外 CDN 能够实时地根据网络流量和各节点的连接、负载状况以及到用户的距离和响应时间等综合信息将用户的请求重新导向离用户最近的服务节点上。
-  缓存：将长时间不会改变的第三方类库或者静态资源设置为强缓存，将 max-age 设置为一个非常长的时间，再将访问路径加上哈希达到哈希值变了以后保证获取到最新资源，好的缓存策略有助于减轻服务器的压力，并且显著的提升用户的体验
-  gzip：开启 gzip 压缩，通常开启 gzip 压缩能够有效的缩小传输资源的大小。
-  http2：如果系统首屏同一时间需要加载的静态资源非常多，但是浏览器对同域名的 tcp 连接数量是有限制的(chrome 为 6 个)超过规定数量的 tcp 连接，则必须要等到之前的请求收到响应后才能继续发送，而 http2 则可以在多个 tcp 连接中并发多个请求没有限制，在一些网络较差的环境开启 http2 性能提升尤为明显。
-  懒加载：当 url 匹配到相应的路径时，通过 import 动态加载页面组件，这样首屏的代码量会大幅减少，webpack 会把动态加载的页面组件分离成单独的一个 chunk.js 文件
-  预渲染：由于浏览器在渲染出页面之前，需要先加载和解析相应的 html、css 和 js 文件，为此会有一段白屏的时间，可以添加loading，或者骨架屏幕尽可能的减少白屏对用户的影响体积优化
-  合理使用第三方库：对于一些第三方 ui 框架、类库，尽量使用按需加载，减少打包体积
-  使用可视化工具分析打包后的模块体积：webpack-bundle- analyzer 这个插件在每次打包后能够更加直观的分析打包后模块的体积，再对其中比较大的模块进行优化
-  提高代码使用率：利用代码分割，将脚本中无需立即调用的代码在代码构建时转变为异步加载的过程
-  封装：构建良好的项目架构，按照项目需求就行全局组件，插件，过滤器，指令，utils 等做一 些公共封装，可以有效减少我们的代码量，而且更容易维护资源优化
-  图片懒加载：使用图片懒加载可以优化同一时间减少 http 请求开销，避免显示图片导致的画面抖动，提高用户体验
-  使用 svg 图标：相对于用一张图片来表示图标，svg 拥有更好的图片质量，体积更小，并且不需要开启额外的 http 请求
-  压缩图片：可以使用 image-webpack-loader，在用户肉眼分辨不清的情况下一定程度上压缩图片。

## 17、Vue中的Key的作用是什么？

key 的作用主要是为了高效的更新虚拟 DOM。要解释 key 的作用，不得不先介绍一下虚拟 DOM 的 Diff 算法了。
我们知道，vue 和 react 都实现了一套虚拟 DOM，使我们可以不直接操作 DOM 元素，只操作数据便可以重新渲染页面。而隐藏在背后的原理便是其高效的 Diff 算法。vue 和 react 的虚拟 DOM 的 Diff 算法大致相同，其核心有以下两点：
- 两个相同的组件产生类似的 DOM 结构，不同的组件产生不同的 DOM 结构。
- 同一层级的一组节点，他们可以通过唯一的 id 进行区分
当页面的数据发生变化时，Diff 算法只会比较同一层级的节点
- 如果节点类型不同，直接干掉前面的节点，再创建并插入新的节点，不会再比较这个节点以后的子节点了。
- 如果节点类型相同，则会重新设置该节点的属性，从而实现节点的更新。

当某一层有很多相同的节点时，也就是列表节点时，Diff 算法的更新过程默认情况下也是遵循以上原则。如果更新列表中间的节点。所以我们需要使用 key 来给每个节点做一个唯一标识，Diff 算法就可以正确的识别此节点，找到正确的位置区插入新的节点，不然就会低效的删除更新后的节点并重新生成。

## 18、webpack和vite构建打包的原理是什么？为什么vite比webpack块？
- Webpack基于打包时构建模式，从入口递归分析所有依赖，使用各种loader处理不同类型文件，将转换后的模块打包成少量bundle文件，代码分割、树摇(Tree-shaking)等，最后输出最终文件。特点是所有模块预先构建，生成最终bundle，热更新时需要重新构建受影响模块，使用内存文件系统加速重建。
- Vite基于使用时按需编译模式，开发环境下，启动开发服务器 → 浏览器请求模块 → 实时转换返回 → 按需编译；生产环境下：先预构建依赖 → 使用Rollup打包 → 优化输出
**为什么Vite比Webpack快** 
- 构建打包上：Webpack所有模块均打包构建，启动前构建完整依赖图，模块内容变化需重新构建；vite则是在使用时按需进行编译，编译浏览器请求的模块。启动时无需等待打包完成，精确定位重编译变化模块 
- ESM原生支持：Webpack需处理所有模块并转换为特定格式，Vite直接利用浏览器原生import，无需处理；
- 预构建优化：Vite聪明地预构建并缓存node_modules依赖；依赖很少变化，构建一次多次复用，转换CommonJS为ESM供浏览器使用，将多个内部模块合并减少请求；
- 热更新实现区别：Webpack HMR需要重新构建整个模块 → 生成更新描述 → 推送给客户端；Vite HMR则直接精准定位变化模块 → 简单转换 → 推送给浏览器重新请求
- 性能对比：Webpack启动时间随项目大小增长，热更新速度受影响模块范围相关，构建慢，增量构建缓解；而vite启动时间基本恒定,热更新速度块；
- 实际场景：Webpack适合需要更强兼容性的项目，大量非ESM模块的项目，复杂定制构建流程；Vite适合现代浏览器为主的项目，快速迭代开发，中大型单页应用


## 19、Cookie，localStorage，sessionStorage的区别？
- 存储的时间有效期不同：cookie的有效期是可以设置的，默认的情况下是关闭浏览器后失效；sessionStorage的有效期是仅保持在当前页面，关闭当前会话页或者浏览器后就会失效；localStorage的有效期是在不进行手动删除的情况下是一直有效的；
- 存储的大小不同：cookie的存储是4kb左右，存储量较小，一般页面最多存储20条左右信息；localStorage和sessionStorage的存储容量是5Mb
- 与服务端的通信：cookie会参与到与服务端的通信中，一般会携带在http请求的头部中，例如一些关键密匙验证等；localStorage和sessionStorage是单纯的前端存储，不参与与服务端的通信
- 读写操作的便捷程度：cookie的相关操作，cookie操作起来较为繁琐，并且部分数据不可以读取操作

## 20、前端常见安全攻击及防御
- XSS攻击(跨站脚本攻击)：XSS是最常见的前端安全漏洞，攻击者通过注入恶意脚本到页面执行；常见类型包含反射型的 XSS 的恶意脚本(存在 URL 里)，存储型 XSS 的恶意代码(存在数据库里) 防御措施：对用户的输入进行前后端过滤；对 HTML 做充分转义；通过HttpOnly Cookie防止JavaScript访问Cookie
- CSRF攻击(跨站请求伪造)：原理：受害者登录A网站，并且保留了登录凭证，攻击者引诱受害者访问B网站，B网站向A网站发送了一个请求，浏览器请求头中会默认携带 A 网站的 Cookie，A 网站服务器收到请求后，经过验证发现用户是登录了的，所以会处理请求。预防措施：CSRF大多来自第三方网站，那么我们就直接禁止第三方域名（或者不受信任的域名）对我们发起请求；请求中包含不可预测的令牌（CSRF Token），限制跨站Cookie发送（SameSite Cookie），检查Referer/Origin（验证请求来源），关键操作二次验证。
- 点击劫持(UI覆盖攻击)：通过透明iframe覆盖目标网站，诱导用户点击。（X-Frame-Options头：禁止iframe嵌套）

## 21、谈谈js的事件循环机制？
JavaScript是单线程语言，这意味着它同时只能执行一个任务。为了处理异步操作，如网络请求、文件读写等，JavaScript使用事件循环机制进行任务调度。在JavaScript中，所有的任务都可以分为
- 同步任务：立即执行的任务，同步任务一般会直接进入到主线程中执行
- 异步任务：异步执行的任务，比如ajax网络请求，setTimeout定时函数等
同步任务进入主线程，即主执行栈，异步任务进入任务队列，主线程内的任务执行完毕为空，会去任务队列读取对应的任务，推入主线程执行。上述过程的不断重复就事件循环。异步任务还可以细分为
- 微任务（Promise.then，MutaionObserver，process.nextTick等）
- 宏任务（setTimeout，postMessage，I/O等）

执行一个宏任务的时候，如果遇到微任务就将它放到微任务的事件队列中，当前宏任务执行完成后，会查看微任务的事件队列，然后将里面的所有微任务依次执行完。

## 23、内存泄漏常见原因及优化方式？
常见内存泄漏原因：
- 全局变量
- 被遗忘的定时器
- 闭包引用
- DOM引用
优化方式：
- 及时清除引用
- 使用WeakMap/WeakSet（只能用对象作为键，引用计数法自动删除）
- 注意销毁事件监听器
- 避免创建不必要对象
